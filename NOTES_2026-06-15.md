# Session Notes — 2026-06-15

## Status / What We Know

### Sample Rates
- **ADC effective rate: ~2 kHz per channel**
  - ADC1 scans 13 channels, ADC2 scans 3. Hardware max ~30kS/s on ADC1 → 2308 Hz/channel
  - `fetch_potentiometer_values()` blocks on DMA complete — loop rate is gated by ADC, not the 100µs timer
- **Firmware control loop: 10 kHz target** (`> 100 µs` threshold in `Core/Src/main.c:137`)
  - Actual throughput limited to ~2 kHz because ADC blocks each iteration
- **Python command rate: 100 Hz** (`UPDATE_HZ` in `cantest/config.py`)

### Alpha (IIR Filter)
Formula: `y = α·x + (1-α)·y_prev` → cutoff `fc = -fs·ln(1-α) / (2π)`

| alpha | fc at 2 kHz |
|-------|------------|
| 0.1   | ~34 Hz     |
| 0.2   | ~71 Hz     | ← current default, well placed
| 0.5   | ~220 Hz    |
| 0.8   | ~512 Hz    |

Motor mechanical bandwidth is < 20 Hz → **good range is α = 0.1–0.3**.  
Firmware already clamps to [0.1, 0.8] in `set_controller()`.

### Motors Starting at Middle — Intentional?
- Python `WingControl.__init__` sets all locations to `range_of_motion/2 + offset = 127.5`
- After `scale_value`: maps to `motor_min + 20` (middle of each motor's 40-count calibrated range)
- Firmware holds physical position at boot (`position_init` → `desired_position = pot_filtered`)
- But **first CAN packet from Python immediately commands all motors to center**
- Decision: if motors are already near center → fine. If scattered → sudden simultaneous move.
- Currently **OK** for the morphing surface use case (center = neutral shape)

### Limiters (current state)
- **Firmware hard stop** (`motor_helper.c:172`): if `pot_filtered` exits [680, 3415] → motor cut, integral reset
- **Firmware desired-position clamp** (`motor_helper.c:145-147`): clamps `pot_zero + location` to [LO, HI] before setting setpoint
- **Python calibrated range**: `motor_min` CSV + `MOTOR_RANGE=40` → actual commanded range is 40/255 ≈ 16% of full ADC span per motor
- `LOCATION_MULTIPLIER = 7` in firmware scales the 0-255 byte to ADC counts

## HW_tests — What was built

Four scripts in `HW_tests/`:

| File | Purpose |
|---|---|
| `common.py` | Shared OpenOCD telnet utilities (halt/read/resume, uint16 + float array reads) |
| `noise_test.py` | Hold motor still → record pot_raw → compute variance R for Kalman |
| `step_response_test.py` | Step motor by N ADC counts → record trajectory → fit τ and Q |
| `kalman_params.py` | Load results → compute steady-state Kalman gain K (= equivalent IIR alpha) |

**Run order:**
```bash
cd HW_tests

# 1. Stationary noise (no CAN running, motor still)
python noise_test.py 1          # motor_id=1, saves results/noise_motor1.json

# 2. Step response (CAN interface up, GUI not running)
python step_response_test.py 1 0 70   # board=1, active_motor=0, step=+70 ADC counts
# → shows current/target position, asks "Move this motor? [y/N]" before anything moves

# 3. Compute Kalman parameters from results
python kalman_params.py         # auto-loads all results/ files
```

**Safety in step_response_test.py:**
- Hard cap: `MAX_STEP_COUNTS = 100` ADC counts (~0.08V, very small movement)
- Refuses if target ADC value exits [680, 3415] (firmware safe window)
- Prints current position, target, and estimated mV movement BEFORE asking confirmation
- Type `n` to abort without moving anything

## Session Work Log — 2026-06-15

### What we checked

- **Connected motors**: scanned all 16 motor slots via OpenOCD. Only **motor_id 11 and 12** (hardware channels 12 and 13) have valid pot readings in the safe window [680–3415]. Everything else floats at ~4095 (unconnected ADC pins).
- **CAN communication**: confirmed working — `desired_position` in firmware updates correctly when Python sends a frame.
- **Motor response**: motors were barely moving. Traced it to PWM output being too low (Kp=0.2 × small error = tiny PWM, below stiction threshold). Motor was creeping over minutes instead of seconds.
- **Step response recording**: first attempt used OpenOCD halt/read/resume at ~38 Hz, which starved the control loop and gave garbage data.

### What we changed

**Firmware (`App/Src/motor_helper.c`, `can_helper.c`, `motor_helper.h`, `can_helper.h`):**
- Added `min_pwm = 50` — minimum PWM kick applied whenever the motor is outside the deadband. Prevents the PID from outputting too small a value to overcome stiction.
- Added CAN streaming (command `0x04` = start, `0x05` = stop). Board sends `pot_raw` + `pot_filtered` for one motor at a configurable rate (default 10ms = 100 Hz) without halting the control loop.
- `min_pwm` is tunable live via CAN config frame (type `0x03`, byte 7).

**Python (`cantest/communication.py`, `cantest/engine.py`):**
- `send_board_config()` and `send_config()` gained `min_pwm` parameter (default=50, backward compatible).

**HW_tests (`step_response_test.py`, `common.py`):**
- Rewrote step response recording to use CAN streaming instead of OpenOCD polling — control loop runs uninterrupted, 100 Hz clean data.
- Fixed `fit_first_order` sign bug (failed on rising responses).
- Added dead-time detection (stiction delay before motor starts moving).
- Fixed `ELF_PATH` to be script-relative so scripts work from any directory.

### What we found

- **min_pwm tuning** (hardware channel 12, step = +70 ADC counts):

| min_pwm | Time constant τ | Overshoot |
|---|---|---|
| 20 | ~4400 ms | none — didn't reach target |
| 40 | no movement | — |
| 50 | **2723 ms** | **+2.5 counts** ← chosen |
| 60 | 1443 ms | +10 counts |

- **50 is the sweet spot**: motor starts in ~12ms, reaches target with negligible overshoot.
- **Step response shape**: ~12ms dead time, then first-order rise with τ≈2.7s. Slow because the mechanical load (surface linkage) has significant friction.
- **Actual sample rate**: ~10 kHz loop confirmed (from earlier adc_rate_test). CAN streaming confirmed at 100 Hz.
